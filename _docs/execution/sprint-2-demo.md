# Sprint 2（第2个月）评测稳定化 + Android 并行接入

## 目标

- 建立稳定评测节奏（训练后固定回归）。
- 打通 Android 最小可用主链路：随手记 -> 头脑风暴 -> 卡片收成。

## 时间与投入

- 周期：4 周
- 预计投入：约 80 小时（每周 20 小时）
- 预算建议：120-180 GPU 小时中的 35-55 小时用于本 Sprint 微调迭代

## Must / Should / Won't

- Must
  - 固化训练后自动回归流程（以 Layer 2 为主）。
  - 完成 1-2 轮保守微调迭代，优先修复 Sprint 1 暴露问题。
  - Android 接入模型推理能力，跑通最小主链路。
  - 对齐数据流与隐私开关的最小可用行为（授权后入池、关闭后不入池）。
- Should
  - 建立简单的异常输入测试集合（空输入、超长输入、特殊字符）。
  - 输出设备侧性能观察（首 token 延迟、完整响应时间）。
- Won't
  - 不扩展 Stage 2 个性化。
  - 不引入多模态与复杂工具调用。

## 每周拆分

### Week 1：评测流水线稳定化

- 训练后自动触发 Layer 2 回归。
- 固化评测结果存档结构，确保可追溯。

交付物：
- `s2-eval-pipeline-v1`
- `s2-eval-archive-structure`

### Week 2：保守微调迭代

- 执行第 1 轮保守迭代，观察核心能力与中文保护题变化。
- 若有明显退化，执行第 2 轮纠偏迭代。

交付物：
- `s2-train-e01` / `s2-train-e02`（按需）
- `s2-regression-compare`

### Week 3：Android 主链路打通

- 完成端侧模型推理接入。
- 实现并验证最小主链路：随手记 -> 头脑风暴 -> 卡片收成。

交付物：
- `android-demo-v0`
- `s2-mainflow-test-record`

### Week 4：Gate2 评审与问题归档

- 汇总评测稳定报告与端侧问题清单。
- 完成 Gate2 评审并给出下一 Sprint 输入。

交付物：
- `s2-gate2-review`
- `s2-known-issues`
- `s2-retro`

## Gate2 通过标准

- Android 主链路可在真实设备跑通。
- 回归评测未触发 P0/P1 红线。
- 至少一版模型可支持 Demo 稳定演示。

## 风险与止损

- Android 集成超时：若 Week 3 仍未跑通，立即冻结可选优化，仅保主链路。
- 评测波动过大：同配置波动影响决策时，优先增加复跑次数而非继续开新实验。
- 中文体验偏弱：保留“可用优先”，在 Sprint 3 再集中优化可读性。

## Sprint 结束复盘模板

- Gate2 是否通过：通过 / 有条件通过 / 未通过
- 主链路稳定性：成功率、异常类型、阻塞点
- 评测趋势：核心能力、中文保护题、格式一致性
- Sprint 3 输入：必须修复项 / 可延后项
