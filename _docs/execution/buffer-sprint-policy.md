# 缓冲 Sprint 触发与执行规则

## 目的

- 当主 Sprint 触发红线或关键门禁未通过时，快速进入“修复模式”。
- 防止范围膨胀，确保节奏可控、成本可控、目标不漂移。

## 缓冲 Sprint 类型

## BufferSprintA（训练修复）

- 适用场景
  - Gate1 未通过。
  - 出现 P0/P1 红线。
  - 同配置复跑波动过大，结论不可判定。
- 允许工作
  - 数据配方修正
  - 回归评估复跑
  - 红线样本清理与根因分析
- 禁止工作
  - 新功能开发
  - 切换主线模型
  - 扩展 Stage 2

## BufferSprintB（Demo 稳定）

- 适用场景
  - Gate2 未通过。
  - Demo 主链路在真实设备不稳定。
  - 中文可读性低于“可用”标准且影响演示。
- 允许工作
  - 稳定性修复（崩溃、超时、异常输入）
  - 中文质量补强（小范围、可复现）
  - 验收脚本与演示路径收敛
- 禁止工作
  - 新增大功能
  - 多模态扩展
  - 工具生态扩展

## 触发条件矩阵

| 条件 | 动作 | 缓冲类型 |
|------|------|----------|
| 触发 P0 安全红线 | 立即冻结主 Sprint，进入修复 | BufferSprintA |
| 触发 P1 功能红线 | 终止当前实验分支，回滚并修复 | BufferSprintA |
| Gate1 未通过 | 停止扩展任务，进入训练修复 | BufferSprintA |
| Gate2 未通过 | 冻结新功能，进入 Demo 稳定修复 | BufferSprintB |
| 连续两周周目标未达成 | 下调范围，仅保 Must 项 | BufferSprintB |

## 执行时长与节拍

- 建议长度：1 个 Sprint（4 周）
- 最长上限：2 个缓冲 Sprint
- 若第 2 个缓冲 Sprint 后仍未通过门禁：
  - 保留演示最小闭环
  - 下调里程碑目标
  - 重排下阶段计划

## 缓冲 Sprint 内部节奏（周视角）

### Week 1：冻结与分诊

- 冻结新需求
- 确认 Top 5 阻塞问题
- 建立修复优先级（P0 > P1 > P2）

### Week 2：集中修复

- 仅处理 Must 修复项
- 每次修复后立即回归验证

### Week 3：复测与收敛

- 聚焦剩余高频问题
- 收敛演示路径（不扩范围）

### Week 4：门禁复评

- 重新评审 Gate1 或 Gate2
- 输出“恢复主 Sprint / 继续缓冲 / 下调目标”结论

## 门禁复评模板

- 复评门禁：Gate1 / Gate2
- 修复清单完成率：x / y
- 红线状态：P0 / P1 / P2
- 决策结论
  - 恢复主 Sprint
  - 继续 Buffer（仅一次）
  - 下调目标并收口

## 预算护栏

- 若累计 GPU 成本达到总预算 80%，立即启用“保核心策略”：
  - 仅保回归评估 + Demo 主链路
  - 暂停探索性训练
  - 暂停非关键体验优化
