# 沟通小结 · 2026-05-02

本文档整理当日与「小模型微调 + 灵感/脑暴产品形态」相关的讨论要点，便于后续接续推进。

---

## 1. 背景与目标

- 对大模型**微调**感兴趣；关注**参数较小的模型**（如 Gemma E2B/E4B、Qwen 3.4 0.8B/2B 等，具体以官方卡与许可证为准）。
- 长期希望能在**端侧或轻量场景**使用；训练环境计划使用 **AutoDL 租赁 GPU** 在云端微调。
- **语言**：中英文混合。
- 项目目的之一：**系统学习如何完成一次微调全流程**（数据 → 训练 → 评测 → 部署/迭代）。

---

## 2. 产品形态演变

### 2.1 早期设想（已调整）

- 曾考虑 **Kotlin + Android Studio** 原生应用，覆盖 Android 与华为鸿蒙；本地部署小模型；功能包括：记录灵感、打标签、相似灵感「图谱式」关联、语音/文字对话、对话与摘要本地保存等。
- 讨论结论：**「图谱」第一版**更现实的做法往往是 **embedding 相似度建边**，不必一上来真做 GNN。

### 2.2 当前共识（重点转移）

- **不再把重心放在长时间开发原生 App**。
- **形态**：**微信小程序** 承载交互（灵感、对话、展示等）；**小模型以 API 方式部署在服务端**，手机端通过 **网络调用**，**不把整模型装进手机**。
- **优点**：开发量小、迭代快、覆盖微信生态内多机型；模型更新无需用户下载大权重；与「周期性用服务端沉淀的对话做微调」天然契合。
- **取舍**：数据与推理在云端，**隐私叙事**从「全本地」变为「上云 + 可控的个性化」；需考虑 API 成本、鉴权、弱网；微信小程序 **AI 类目与审核** 需提前对照平台规则。
- **技术栈备注**：若未来仍做原生，Kotlin 方案仍可作为扩展方向，但当前阶段不作为主线。

## 3. 微调策略：两阶段划分

### 第一阶段——任务向：强化 Brainstorming / 提炼灵感

- **动机**：小模型本身对「帮用户脑暴、提炼灵感」没有强倾向，可用公开数据做 **SFT/LoRA** 强化该能力。
- **数据倾向**：认可使用 **`brainstorm_vicuna_10k`**（Hugging Face：`DevQuasar/brainstorm_vicuna_10k`）等**头脑风暴、人机对话**类数据；同系列还有带 Markdown 小结的版本（如 `brainstorm-v3.1_vicnua_1k`）更贴近「对话后整理」。
- **注意**：该类数据多为**英文、合成**；若产品主场景为中文，需规划 **中英混合或补充中文脑暴/摘要数据**，避免能力偏科。
- **输出形态**：建议尽早统一——自由对话为主，或固定 **标题/要点/标签** 等结构化输出，以便与小程序落库字段一致。

### 第二阶段——用户向：个性化（「越来越懂你」）

- **思路**：在真实使用一段时间后，用 **用户在小程序中产生的高质量记录**（对话、已采纳的灵感摘要等）做 **周期性微调**（例如每两周或累积到 N 条再训），强化语气、追问习惯、摘要风格等。
- **工程上建议**：任务 LoRA 与个人 LoRA **分权重管理**，便于单独回滚「个人层」而不影响脑暴底座；训练时 **混入通用高质量数据** 缓解灾难性遗忘。
- **数据质量**：优先 **已采纳摘要、用户标记「好」的轮次、或人工确认批次** 进入训练池，减少噪声。
- **与 RAG**：长期事实、旧灵感更适合 **结构化存储 + 检索**；微调侧重「怎么说、怎么问、怎么总结得像你」。

### 产品原则（当日确认）

- **个性化优先**：默认方向是 **允许用用户数据参与模型改进**（第二阶段微调）。
- **仍建议保留的工程底线**（不改变「个性化优先」，降低风险）：
  - **知情与可控**：说明数据用途；提供 **关闭参与训练** 或 **删除训练用数据** 的入口。
  - **安全**：HTTPS、存储加密、账号与数据隔离。
  - **合规与审核**：微信小程序 AI 相关类目与内容规范需单独核对。

## 4. 「数据少」与公开数据集

- **问题**：个人日常使用产生的**高质量成对数据**有限，单靠自采可能不够一轮强 SFT。
- **策略**：
  - **混合训练**：大量通用指令数据打底 + 少量个人精品数据。
  - **合成数据**：用大模型按固定系统提示生成「碎片 → 追问 → 结构化灵感」样本，再人工抽审。
  - **训练节奏**：不必机械每两周必训；可改为 **累积到 N 条已审核样本** 再训。

## 5. 概念澄清：为何「摘要/改写/翻译」仍可能要做微调

- 基座 **Instruct/Chat** 模型 + Prompt 往往已能完成通用摘要、改写、翻译。
- **微调的价值**更常见于：**固定格式、品牌语气、领域术语、中英混合习惯、蒸馏大模型在窄任务上的行为**，而非「教会模型摘要」本身。
- 与当前项目一致：**第一阶段**显式强化 **脑暴与灵感提炼**；**第二阶段**强化 **个人风格与习惯**。

## 6. 边缘小模型能做什么（供后续选型参考）

- **适合**：短摘要、改写、句级翻译、轻量对话、简单结构化抽取、隐私敏感场景的本地处理（若未来再做端侧）。
- **不适合单独扛**：强事实、实时信息（需检索）、复杂长推理；多模态需另接模块。
