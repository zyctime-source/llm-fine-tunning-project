# 5. 产品形态（Surface）

> 本文档为项目 **shaping** 阶段方案描述：界定客户端形态、推理部署形态及关键决策理由。**不包含**具体技术实现细节、Android 版本兼容性矩阵或应用商店上架时间表。

---

## 5.1 客户端形态（决策已定）

### 5.1.1 首版形态

- **形态**：Kotlin 原生 Android 应用
- **排除选项**：微信小程序、纯 Web、Kotlin Multiplatform 跨平台

### 5.1.2 决策理由

| 考量维度 | Kotlin 原生 Android | 微信小程序 | 纯 Web / PWA |
|----------|---------------------|------------|--------------|
| **端侧模型部署** | ✅ 完全可控，本地推理可行 | ❌ 几乎不可行，主要依赖云端 API | ⚠️ WebAssembly/WebGPU 新兴，兼容性待验证 |
| **隐私与离线** | ✅ 数据可不出端，支持离线使用 | ❌ 数据需上传，依赖网络 | ⚠️ 离线能力有限 |
| **系统级能力** | ✅ 通知、快捷输入、后台服务 | ❌ 受限 | ❌ 受限 |
| **全球推广** | ✅ Google Play 覆盖全球 | ❌ 微信生态主要服务中文用户 | ✅ 浏览器全球可用 |
| **开发者经验** | ✅ 已有 Kotlin 经验，学习曲线平缓 | ⚠️ 需学习小程序框架 | ⚠️ 端侧推理技术栈不同 |
| **性能** | ✅ 最优 | ⚠️ 受限于微信 Runtime | ⚠️ 受限于浏览器性能 |

**核心决策逻辑**：
- 本地模型部署是微调项目的核心目标，小程序/纯 Web 难以满足
- 全球可用性（Google Play），不受单一生态锁定
- 开发者已有 Kotlin 经验，PoC 风险可控

### 5.1.3 iOS 策略

- **首版范围**：不包含 iOS 客户端
- **日后评估**：若需要 iOS 版本，独立评估以下路线：
  - **路线 A**：Swift 原生开发（体验最优，两套代码维护）
  - **路线 B**：届时 Kotlin Multiplatform / Compose Multiplatform 成熟度再评估代码共享

## 5.2 推理部署形态（端云协同）

### 5.2.1 端侧推理（主要形态）

- **定位**：默认启用，提供「开箱即用」的低延迟体验
- **适用场景**：快速响应、基础脑暴、隐私敏感、离线可用
- **模型规模**：小参数模型（0.5B - 3B）
- **技术路径**：GGUF 格式 via llama.cpp，或 ONNX via ONNX Runtime，或 TFLite（实现阶段评估）

### 5.2.2 云端推理（备选形态）

- **定位**：显式提供切换入口，用于深度生成与用户对比学习
- **适用场景**：复杂推理、长文本总结、多角度发散
- **候选模型**：DeepSeek、Qwen 等大模型 via API
- **用户价值**：满足「对比微调小模型 vs 大模型默认行为」的学习诉求

### 5.2.3 协同逻辑（shaping 占位）

- 首版 shaping **不定**具体切换策略（自动 fallback / 手动切换 / 智能路由）
- 保留概念位：用户可感知「端侧快响应」与「云端深能力」两种模式
